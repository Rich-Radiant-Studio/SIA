# 贡献指南

## 开发哲学

本项目遵循**保守工程原则**：

> "稳定 > 创新，简单 > 复杂，最小化 > 最大化"

## 提交代码前的检查清单

### 必须检查：
- [ ] 我的改动是否必要？
- [ ] 我的改动是否最小化？
- [ ] 我是否使用了现有的技术栈？
- [ ] 我是否保持了代码风格一致？
- [ ] 我是否引入了新的依赖？（如果是，为什么？）
- [ ] 我是否改变了现有架构？（如果是，为什么？）
- [ ] 我的改动是否经过测试？

### 代码审查标准

#### 会被拒绝的 PR：
- ❌ 不必要的重构
- ❌ 引入新的框架/库（未经讨论）
- ❌ 改变现有架构（未经讨论）
- ❌ 大规模的代码重组
- ❌ 改动范围过大
- ❌ 缺少必要的说明

#### 会被接受的 PR：
- ✅ 最小化的改动
- ✅ 清晰的改动说明
- ✅ 保持代码风格一致
- ✅ 使用现有技术栈
- ✅ 充分的测试
- ✅ 向后兼容

## 提交信息规范

### 格式：
```
<type>: <简短描述>

详细说明：
- 为什么需要这个改动
- 改动了什么
- 影响范围
- 测试情况
```

### 示例：
```
fix: 修复登录按钮样式问题

详细说明：
- 登录按钮在移动端显示不正确
- 修改了 Login.vue 中的按钮样式
- 只影响登录页面
- 已在 Chrome/Safari/Firefox 测试
```

## 开发流程

### 1. 开始开发前
```bash
# 1. 拉取最新代码
git pull origin main

# 2. 创建功能分支
git checkout -b fix/button-style

# 3. 确认改动必要性
# 问自己：这个改动真的必要吗？
```

### 2. 开发过程中
```bash
# 1. 只修改必要的文件
# 2. 保持改动最小化
# 3. 频繁提交（小步快跑）
git add <changed-files>
git commit -m "fix: 修复按钮样式"
```

### 3. 提交 PR 前
```bash
# 1. 检查改动范围
git diff main

# 2. 确认没有不必要的改动
# 3. 运行测试
npm run test

# 4. 推送代码
git push origin fix/button-style
```

## 代码风格

### 遵循现有风格
```typescript
// ✅ 好：遵循现有风格
const userName = 'John'  // 项目使用 camelCase

// ❌ 坏：引入新风格
const user_name = 'John'  // 不要使用 snake_case
```

### 保持一致性
```vue
<!-- ✅ 好：与现有组件保持一致 -->
<template>
  <div class="container">
    <!-- 使用项目已有的类名 -->
  </div>
</template>

<!-- ❌ 坏：引入新的命名方式 -->
<template>
  <div class="wrapper">
    <!-- 不要随意改变命名 -->
  </div>
</template>
```

## 技术选型

### 决策流程
```
需要添加新功能
    ↓
现有技术能实现吗？
    ↓ 是
使用现有技术 ✅
    ↓ 否
团队熟悉的技术能实现吗？
    ↓ 是
使用团队熟悉的技术 ✅
    ↓ 否
成熟稳定的技术能实现吗？
    ↓ 是
讨论后使用成熟技术 ⚠️
    ↓ 否
重新评估需求 ❌
```

## 常见问题

### Q: 我发现代码可以优化，应该重构吗？
**A:** 不应该。除非：
- 影响功能正常运行
- 有明确的性能问题
- 团队一致同意

### Q: 我想引入一个新的库，可以吗？
**A:** 需要讨论。问自己：
- 现有技术真的不能实现吗？
- 这个库足够成熟吗？
- 团队成员熟悉吗？
- 维护成本如何？

### Q: 我的 PR 改动很大，怎么办？
**A:** 拆分成多个小 PR：
- 每个 PR 只做一件事
- 每个 PR 都可以独立审查
- 每个 PR 都可以独立回滚

### Q: 代码风格不统一，应该统一吗？
**A:** 不应该大规模修改。应该：
- 新代码遵循现有风格
- 修改代码时顺便调整
- 不要为了风格而修改

## 记住

**我们的目标是：**
- 保持代码库稳定
- 降低维护成本
- 提高代码可预测性
- 减少意外问题

**我们不追求：**
- 最新的技术
- 最完美的代码
- 最优的性能
- 最酷的实现

---

**最重要的一句话：**

> "如果不确定是否应该做某个改动，那就不要做。"
